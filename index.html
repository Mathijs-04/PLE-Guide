This is a guide for making your own AI Rule Assistant

This page is written for builders who want to create a reliable AI assistant for a rules-heavy domain: tabletop games, school policies, internal knowledge bases, compliance documents, or any system where answers need to be traceable.

I built the Warhammer Rule Assistant as a working example, then turned the lessons from that project into a practical guide. The goal is to share the architecture, design choices, tips, and best practices that made the assistant more trustworthy than a standard chatbot.

What we are building

A reliable AI rule assistant

Not just a chatbot

A rule assistant must answer from an approved knowledge base, show where the answer came from, and avoid pretending that uncertain interpretations are facts.

Rules are unusually hard

They include exceptions, timing windows, nested definitions, conflicting interpretations, and frequent updates. A confident but unsupported answer can be worse than no answer.

Reliability beats speed

The goal is not the fastest response. The goal is a response that users can inspect, verify, and challenge when the rules are ambiguous.

The problem

Why standard LLMs fail at rule systems

Large language models are excellent at language, but rules-heavy domains need more than fluent text.

01

Hallucinations look convincing

An LLM may invent a rule, merge old editions, or apply a similar rule from another domain unless the system constrains its context.

02

Important details hide in edge cases

Rule answers often depend on exact timing, exceptions, keywords, source hierarchy, or a single sentence in a faction-specific entry.

03

Users need proof, not vibes

Without source attribution and certainty language, users cannot tell whether an answer is directly supported, interpreted, or barely covered.

High-level architecture

Query, retrieve, ground, answer

The Warhammer assistant uses Laravel and Vue for the interface, then sends questions to a Python RAG service. The same pattern works for any domain with reliable source documents.

1

User query

The user asks in natural language and selects the relevant ruleset or domain.

2

Retrieval system

The backend searches indexed source material for the most relevant rules.

3

Context selection

Semantic chunks, keyword matches, and concept sections are combined into a compact prompt.

4

LLM response

The model answers only from the supplied context and follows a strict response format.

5

Structured answer

The UI shows a short answer, detailed explanation, sources, and certainty level.

  • Frontend: Vue chat interface with a Questions and Core Rules navigation pattern.
  • API: Laravel accepts the question at /api/chat and proxies it to Python.
  • RAG: FastAPI answers through FAISS indexes built from markdown rule sources.
  • Trust: Responses include structured JSON, source labels, certainty, and PDF deep links.

Vector-based retrieval

RAG makes the model read before it speaks

Retrieval-Augmented Generation gives the assistant a focused packet of relevant source material instead of asking the model to rely on memory.

Chunk the knowledge

Source documents are split into meaningful sections. In the Warhammer assistant, markdown rules are chunked by headings and stored with metadata such as game system, faction, and document type.

Embed each chunk

Embeddings turn text into numbers that capture meaning. Similar rule concepts end up close together even when the user uses different wording.

Search by similarity

A vector index finds chunks that are close to the user question. Similarity search improves relevance because the model sees the likely rule text before generating an answer.

Add precise retrieval paths

The example project also uses keyword and heading search, fuzzy matching for named rules or units, and concept injection for timing, phase, and sequencing questions.

Structured output

Make every answer inspectable

A reliable assistant should not return one large paragraph. It should separate the conclusion, reasoning, source trail, and uncertainty.

Certainty scale

Certainty is not absolute truth. It is a trust-calibration signal based on source strength, consistency, and coverage.

  1. 1 Directly supported by the retrieved rules.
  2. 2 Mostly supported, but some interpretation is needed.
  3. 3 Barely covered or potentially contradictory.
  4. 4 Not covered well enough to answer confidently.

Source attribution and trust

Traceability turns an answer into something users can verify

Source panels

Show which rulebooks, policies, lessons, standards, or manuals contributed to the answer. Users should know what the model relied on.

Deep links

When possible, link rule terms or answer sections back to the exact PDF page, article, or document section. Verification should be one click away.

Explicit uncertainty

Use certainty scoring to separate directly supported answers from interpreted ones. This prevents the assistant from sounding equally confident about everything.

Design principles

Rules for building rule assistants

01

Transparency over black-box answers

Make the retrieval process visible through sources, quotes, and explanation structure.

02

Traceability of information

Every confident answer should point back to the documents that support it.

03

Trust calibration

Users should understand when the system is certain, interpreting, or unable to answer.

04

Consistent output

Structured responses make the UI predictable and make backend validation easier.

Where this applies

The same pattern works beyond tabletop games

Tabletop and roleplaying games

Handle editions, factions, rule exceptions, scenario packs, errata, and timing disputes with source-linked answers.

Education systems

Guide students through course rules, grading rubrics, institutional policies, and assignment requirements without inventing exceptions.

Operational knowledge assistants

Support teams can apply the same design to product manuals, compliance policies, onboarding material, or internal process documentation.

Key takeaways

Reliability is a system design problem

A strong model helps, but the surrounding architecture decides whether users can trust the answer.

  • RAG improves relevance by retrieving domain-specific context before generation.
  • Sources improve trust because users can verify the answer themselves.
  • Confidence scoring improves understanding by showing how strongly the rules support the conclusion.
  • Structured output makes the assistant easier to test, render, and reason about.
